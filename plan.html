1. System Overview

The engine evaluates 50+ stocks every market hour against AI-generated Python indicator formulas and fires a Discord webhook when a stock crosses its threshold. All computation runs inside a single Docker container.

The architectural goals are:

  • Process isolation — untrusted Python formulas cannot crash the Go orchestrator
  • Decoupled stages — each pipeline stage communicates only via typed Go channels behind defined interfaces; any stage can be replaced independently
  • Formula Manager — dedicated stage between sync and execution; owns formula loading, caching, hot-reload, grouping by FormulaID, and params injection; uses FormulaID + Version for formula identification and state invalidation; no other stage touches formula files
  • Centralized Error Manager (Alerting and logging) — single goroutine receives all error events from all stages; owns burst suppression, routing, and Discord error delivery; stages call Report() and forget
  • Engine Monitor (System-wide health and visibility) — dedicated Engine Monitor receives heartbeats and status updates from all managers, providing a global health view without coupling manager internals together
  • System observability — a dedicated Engine Monitor receives heartbeats and status updates from all managers; tracks queue depths, worker health, and heartbeat freshness; exposes /health and /status endpoints; provides a global health view without coupling manager internals together
  • Resource governance — a central Process Data Manager bounds concurrent Python evaluations and tracks resource usage
  • RAM as serving layer, SQLite as persistence — 2 years of OHLCV in SQLite; ring buffers in RAM serve all evaluations; SQLite read only on startup warm-up
  • Python runtime reuse — subprocesses stay alive across evaluations; pandas/numpy loaded once; per-ticker state preserved for incremental computation; batch execution within same formula group
  • Dual-use base — pipeline interfaces are generic; swapping implementations targets both trading signals and DOMjudge-style code judge systems
Scope note This is an hourly-resolution signal system, not a sub-second HFT engine. Latency target: all 50 stocks evaluated and Discord fired within the same minute the new candle closes.

2. Pipeline Architecture

Each stage is a goroutine (or small fixed pool) that blocks on its input channel and writes to its output channel. No stage holds a reference to another stage's internals. All errors are forwarded to the centralized Error Manager — stages never handle Discord error delivery themselves.

[ Stage 1: Filler API ] │ │ chan WriteRequest ──────────────────────────────► [ SQLite Writer ] │ chan TickerUpdate ▼ [ Stage 2: Sync & Decision ] ◄── reads from ring buffer (RAM) │ │ chan EvalRequest { Ticker, Data } — no formula yet ▼ [ Stage 2.5: Formula Manager ] │ resolves formula, attaches code+hash+params, groups by hash │ │ chan BatchEvalRequest { FormulaHash, Code, []BatchItem } ▼ [ Stage 3: Process Data Manager ] │ │ dispatches to worker pool (semaphore-bounded) ├──► [ PyWorker 0 ] (persistent Python subprocess) ├──► [ PyWorker 1 ] └──► [ PyWorker N ] │ │ chan EvalResult ▼ [ Stage 4: Result Sink ] │ ├──► signal: dedup check → Discord signals webhook └──► error: ──────────────────────────────────────────► [ Error Manager ] ▲ all other stages ────────────────────┘ Report(ErrorEvent) — non-blocking [ Engine Monitor ] ◄── status updates ── all managers │ ├─ manager health & heartbeats ├─ queue depth metrics ├─ worker status └─ /health + /status endpoints

Stage 1 — Filler API Go

Renamed from "API Puller" — the Filler is a continuous priority-queue scheduler, not a fixed hourly batch. Each ticker has a dynamic priority score and a hard cooldown floor (max 1 hour). High-priority tickers get pulled on 15-min live candles; normal-priority tickers get pulled once per hour. Full detail in Section 3 — Filler API.

Responsibilities

  • Maintain a min-heap of tickers ordered by NextEligibleAt + PriorityScore
  • Run a scheduler goroutine that pops eligible tickers, dispatches to a bounded Alpaca fetch pool, recomputes priority from the latest EvalResult.NewState, and pushes back
  • Hourly background path: fetch 1-hour bar → append to RAM ring buffer + SQLite Writer
  • Hot path (high-priority tickers): fetch latest 15-min bar → update liveCandle[ticker] in RAM only — no SQLite write
  • On Alpaca failure: emit TickerUpdate with Err set; Stage 2 skips this ticker for the current evaluation

Tick Alignment & DST

Hourly background ticks fire at HH:02 Eastern time — not UTC. US markets run on Eastern time; using UTC truncation silently misaligns by 1 hour during DST transitions. Compute next tick in Eastern time explicitly. If Alpaca returns empty data (weekend, holiday), the puller emits nothing for that ticker. No external calendar needed.

Stage 2 — Sync & Decision (Time Gate) Go

Collects all TickerUpdates for the current evaluation cycle. Once the full set has landed (or a timeout expires), it performs a staleness check, reads each data array from the RAM ring buffer, and emits EvalRequests to the PDM. SQLite is not touched here.

Time Gate

Stage 2 holds a counter per cycle. When the count reaches len(tickers), or a gate timeout fires (e.g. 3 minutes), it releases — whichever comes first. Tickers that reported errors are excluded from this cycle's evaluation batch.

Why batch-gate instead of per-ticker fire Batch-gating gives a clean synchronization point, makes logging coherent ("tick 14:00 — 49/50 stocks evaluated"), and avoids a race where Stage 3 is still processing last cycle's stocks when new ones arrive.

Staleness Check

Before slicing the ring buffer for a ticker, verify ringBuffer[ticker].Head().Timestamp equals the expected tick timestamp. If it doesn't (Filler write failed silently or Alpaca returned stale data), skip this ticker and log to the error channel. Evaluating stale data produces a silent wrong result — worse than no result.

Array Build

Slices the last N candles from the RAM ring buffer, then appends liveCandle[ticker] as the current tip (if a 15-min bar is available). N is resolved by the lookback cascade:

PrioritySourceExample
1 (highest)Per-ticker override configTSLA → 500 hours
2Formula's declared warm-up requirementEMA-200 → 200 hours
3 (lowest)Global default720 hours (~1 month)

Stage 2.5 — Formula Manager Go

Sits between Sync and PDM. Receives bare EvalRequest{Ticker, Data}, resolves the formula for each ticker, attaches code + FormulaID + Version + params, then groups requests by FormulaID into batches. No other stage reads formula files.

  • Maintains an in-memory formulaCache — file reads happen only on cache miss or file change
  • File-watcher goroutine invalidates cache entries on formulas/*.py modification; new version propagates on next evaluation, triggering prev_state reset automatically
  • Injects per-ticker params from config so tickers can share one formula file with different thresholds/periods
  • Groups tickers sharing the same FormulaID into a single BatchEvalRequest — Python exec()s the formula once for the whole group
  • Fail-open: missing formula file → emit ErrorEvent to Error Manager, skip ticker this cycle — never blocks the remaining batch

Full detail in Section 4 — Formula Manager.

Stage 3 — Process Data Manager (PDM) Go Python

Receives BatchEvalRequests from Formula Manager and dispatches them to a fixed pool of Python workers bounded by a semaphore. Each batch is processed by one worker — the formula is exec()ed once, all items in the batch evaluated sequentially in the same namespace. State reset on formula version mismatch is handled automatically.

Full detail in Section 8 — Process Data Manager.

Stage 4 — Post Processor Go

Receives EvalResults from the PDM's output channel. For results with Signal == 1, applies deduplication logic before firing Discord.

Responsibilities

  • Read last_signal from SQLite for the ticker; skip if within cooldown window
  • POST to Discord webhook with structured embed (ticker, formula ID, version, timestamp, signal value)
  • On Discord success: update last_signal for that ticker via SQLite Writer
  • On Discord failure: retry with exponential backoff (max 3 retries); if still failing, emit to error channel (separate webhook) — never drop silently
  • Error results from PDM (Python timeout, crash): route to error channel, not signal channel

Two-Webhook Pattern

Signal webhook and error webhook are separate Discord channels. An error storm (bad formula update affecting 40 stocks) floods #engine-errors, leaving #signals clean.

3. Filler API — Priority Scheduler

The Filler replaces a naive fixed hourly batch with a continuous priority-queue scheduler. Every ticker has a NextEligibleAt (hard cooldown floor, max 1 hour) and a PriorityScore (soft ordering among eligible tickers). The scheduler goroutine pops, fetches, recomputes, and re-enqueues in a tight loop.

Priority Queue Model

type TickerJob struct {
    Ticker          string
    NextEligibleAt  time.Time   // hard cooldown — never pull before this
    PriorityScore   float64     // lower = pull sooner among eligible set
    LiveTimeframe   string      // "1Hour" | "15Min" — set per priority tier
}
// min-heap: ordered by NextEligibleAt first, PriorityScore as tiebreaker

Scheduler loop:

for {
    job := heap.Peek()
    if job.NextEligibleAt.After(now()) {
        sleepUntil(ctx, job.NextEligibleAt)
        continue
    }
    heap.Pop()
    go fetch(job, alpacaPool, ringBuffers, writerCh, tickerUpdateCh)
    job.NextEligibleAt = now().Add(computeCooldown(job))
    job.PriorityScore  = computePriority(job, ringBuffers, lastResults)
    heap.Push(job)
}

computePriority() reads directly from the RAM ring buffer and the last EvalResult.NewState — zero SQLite I/O:

Priority InputSourceExample
Threshold proximityEvalResult.NewState (e.g. last RSI value)RSI=32 vs threshold 30 → high priority
Recent volatilityRing buffer last 5 closes — stddevHigh stddev → shorter cooldown
Manual tierConfig overrideTSLA.priority = high
Time since last signallast_signal table (cached in memory)Recent signalers monitored more closely

Priority feeds back into cooldown assignment:

ConditionAssigned CooldownFetch Timeframe

High recent volatility20 min15Min live candle
Normal60 min (max)1Hour bar

Two Fetch Modes

Same Alpaca endpoint, different timeframe param. The distinction determines write destination:

[ Hourly background path ] (all tickers, once per hour max) Alpaca timeframe=1Hour │ ├──► ring buffer[ticker].Append(candle) // RAM, immediate └──► SQLite Writer channel // async persist Then emits TickerUpdate → Stage 2 [ Hot path ] (high-priority tickers only, up to every 15 min) Alpaca timeframe=15Min, limit=1 │ └──► liveCandle[ticker] = candle // RAM only, no SQLite write Then emits TickerUpdate → Stage 2

Stage 2 builds the eval array as ring_buffer.Slice(N−1) + liveCandle[ticker]. Historical hourly data is the warm-up; the 15-min tip is the real-time signal surface. Historical bars are treated as constant — the system never re-fetches or reprocesses them.

15-min vs 5-min Decision

Decision: 15-min, configurable per tier Use 15-min as the default live timeframe. 5-min adds noise without meaningful benefit — indicators are computed on hourly history, so a 5-min tip represents 1/12th of the intended data resolution. Sub-15-min price spikes that reverse within the same 15-min window generate false positives. 5-min bars also carry thinner liquidity on low-volume stocks. The sweet spot is 15-min: catches threshold breaches up to 45 minutes earlier than waiting for the hourly close, with acceptable noise. Per-ticker override allows 5-min for high-conviction tickers if needed.
Volume caveat for sub-hourly bars A 15-min bar's volume is not comparable to the hourly bars preceding it in the series. Volume-based indicators (OBV, VWAP, MFI) will see the last element as an outlier by construction. Document this in the formula contract: data[-1] may be a sub-hourly bar; price fields are valid for threshold detection, volume is not normalized.

Alpaca Rate Budget — Three Consumers

The Filler now has three distinct consumers of Alpaca API quota that must not compete:

ConsumerPatternBudget Priority
Backfill workerBulk paginated historical, one-time per tickerLowest — runs in background
Hourly background path50 tickers × 1 request/hourHigh — reserved capacity
Hot path (15-min)Up to N priority tickers × 4 requests/hourHigh — reserved capacity

Use a two-lane rate limiter. Live fetches (hourly + hot path) share a reserved lane with guaranteed capacity. The backfill worker gets a separate, lower-ceiling limiter and consumes only what the live lane leaves unused:

type FillerRateLimiter struct {
    live     *rate.Limiter   // hourly + hot path; reserved capacity
    backfill *rate.Limiter   // lower ceiling, runs opportunistically
}

4. RAM Layer — Ring Buffer & liveCandle

SQLite is the persistence layer. RAM is the serving layer. Stage 2 never reads from SQLite during evaluation — all OHLCV data served to Python comes from in-memory structures. SQLite is only read on startup to warm those structures.

Memory footprint is negligible. 50 tickers × 720-candle ring buffer × 6 float64 fields × 8 bytes ≈ 1.7 MB. Even a full 2-year buffer per ticker is only ~42 MB total.

Ring Buffer Design

type RingBuffer struct {
    data     []OHLCV
    head     int        // next write position
    size     int        // current fill level
    capacity int        // max candles held (= max lookback for this ticker)
    mu       sync.RWMutex
}

func (r *RingBuffer) Append(c OHLCV) {
    r.mu.Lock(); defer r.mu.Unlock()
    r.data[r.head] = c
    r.head = (r.head + 1) % r.capacity
    if r.size < r.capacity { r.size++ }
}

func (r *RingBuffer) Slice(n int) []OHLCV {
    r.mu.RLock(); defer r.mu.RUnlock()
    // returns a copy of the last n candles in chronological order
    // caller owns the slice — no data race on downstream use
}

The mutex lives inside RingBuffer. Slice() returns a copy, not a pointer into the internal array — the EvalRequest data slice is safe to use after the lock is released.

Ring buffer capacity must equal the ticker's max lookback Size each ring buffer to the largest possible lookback for that ticker (from the override cascade). If a formula later requests more history than the buffer holds, it silently receives fewer candles than expected. Crash loudly on startup if capacity < configured_lookback.

liveCandle Map

liveCandle holds the latest 15-min bar per ticker. It is ephemeral — populated by the hot-path fetch, cleared on hour boundary, never written to SQLite.

type FillerState struct {
    RingBuffers map[string]*RingBuffer  // hourly history, persisted to SQLite
    LiveCandle  map[string]OHLCV          // latest 15-min bar, ephemeral
    mu          sync.RWMutex              // guards both maps
}

Stage 2 reads both under a read lock. The Filler hot path writes LiveCandle under a write lock. The hourly background path writes RingBuffers under a write lock.

Hour Boundary Promotion

When the hourly bar closes, two things must happen atomically: append the completed candle to the ring buffer AND clear the liveCandle. If they're not atomic, Stage 2 can briefly see a ring buffer with the new hourly tip AND a stale liveCandle — double-counting the same period.

func (fs *FillerState) PromoteHourly(ticker string, candle OHLCV) {
    fs.mu.Lock()
    defer fs.mu.Unlock()
    fs.RingBuffers[ticker].AppendNoLock(candle)  // inner lock already held
    delete(fs.LiveCandle, ticker)                  // clear atomically
    // SQLite write dispatched separately via writerCh (non-blocking)
}

On container restart: ring buffers are rebuilt from SQLite on startup. LiveCandle is always empty on boot — the hot-path fetch repopulates it within the first 15-minute window. prev_state for Python workers is also lost on restart; formulas fall back to full historical recomputation on the first tick.

5. Python Worker Subsystem

Subprocess Lifecycle Python

Each PyWorker is a Go struct owning one goroutine and one long-running Python subprocess. The subprocess starts when the worker is created at engine boot and runs for the lifetime of the container. It is only restarted if it exits unexpectedly.

[ Go PyWorker goroutine ] │ │ os/exec — one subprocess, alive for container lifetime ▼ [ Python subprocess ] pandas, numpy, pandas_ta loaded once at startup stdin/stdout loop — reads one JSON job, writes one JSON response, repeats

The Python main loop is minimal:

import sys, json, traceback
import pandas as pd
import numpy as np
import pandas_ta as ta

_state_store = {}  # ticker → prev_state dict, persists across calls

for line in sys.stdin:
    job = json.loads(line)
    try:
        result = _run_job(job)
        sys.stdout.write(json.dumps(result) + "\n")
    except Exception as e:
        sys.stdout.write(json.dumps({"error": traceback.format_exc()}) + "\n")
    sys.stdout.flush()
Why persistent subprocess wins here Python cold-start with pandas + numpy + pandas_ta takes 400–800 ms. For 50 stocks per hour, spawning a fresh process per evaluation would consume 20–40 seconds of the hour just on interpreter startup. A persistent subprocess amortizes that cost to zero after boot.

Formula Contract & Formula Storage

Formula Storage — formulas/ Directory

Formulas live in a formulas/ directory mounted at container runtime. One .py file per ticker: AAPL.py, TSLA.py, etc. Each formula file declares a FormulaMeta with an ID and integer Version. Go reads the file content at evaluation time, extracts the meta, and sends both in the EvalRequest. A file-watcher goroutine detects changes and invalidates the cached entry for that ticker — triggering prev_state reset on the next evaluation when the version changes. Formula updates take effect without container restart.

// Mount at runtime: -v ./formulas:/app/formulas
// Each formula file declares its meta:
type FormulaMeta struct {
    ID      string   // e.g. "rsi-mean-revert"
    Version int      // incremented on each change
}
// Go reads formula per evaluation, extracts meta for state invalidation

Every AI-generated formula must expose a single entry point function. Go and Python agree on this interface over JSON via stdin/stdout.

Request Schema (Go → Python stdin)

{
  "ticker":       "AAPL",
  "formula_meta": { "id": "rsi-mean-revert", "version": 3 },  // used for state invalidation
  "formula_code": "def evaluate(df, prev): ...",
  "data": [
    { "t": 1718000000, "o": 180.1, "h": 181.0, "l": 179.5, "c": 180.8, "v": 123456 },

}

Response Schema (Python stdout → Go)

{
  "signal":    1,         // 1 = trigger, 0 = hold
  "new_state": { ... },  // whatever the formula wants to persist
  "error":     null       // non-null string on exception
}

Minimal valid formula

import pandas_ta as ta

def evaluate(df, prev):
    df.ta.rsi(length=14, append=True)
    latest_rsi = df["RSI_14"].iloc[-1]
    signal = 1 if latest_rsi < 30 else 0
    return signal, {"last_rsi": latest_rsi}   # (signal, new_state)

Incremental State Reuse

Because Python subprocesses are persistent and each worker goroutine holds a map[ticker]prevState, formulas can skip recomputing the full historical series each tick. They receive prev_state containing whatever they stored last tick — previous indicator values, partial sums, fitted parameters, etc.

A formula using incremental EMA:

def evaluate(df, prev):
    alpha = 2 / (200 + 1)
    last_close = df["c"].iloc[-1]
    if "ema" not in prev:
        ema = df["c"].ewm(span=200, adjust=False).mean().iloc[-1]
    else:
        ema = alpha * last_close + (1 - alpha) * prev["ema"]   # O(1) update
    return (1 if last_close > ema else 0), {"ema": ema}

State Invalidation on Formula Change

Go stores the FormulaMeta.Version alongside prevState. On each evaluation, if the version in the EvalRequest differs from the stored version, Go sends prev_state: {} and resets the stored version. The formula detects the empty state and falls back to full computation. No stale state leaks across formula versions.

Formula Isolation

Each formula is exec()-ed into a fresh isolated namespace dict on every invocation. This prevents symbol leakage between consecutive stock evaluations inside the same subprocess.

def _run_job(job):
    ticker = job["ticker"]
    formula_meta = job["formula_meta"]

    # State invalidation on formula version change
    stored = _state_store.get(ticker, {})
    if stored.get("_version") != formula_meta["version"]:
        stored = {}

    prev_state = stored.get("state", {})

    df = pd.DataFrame(job["data"])
    df.rename(columns={"t":"timestamp","o":"open","h":"high","l":"low","c":"close","v":"volume"}, inplace=True)

    # Fresh namespace per call — no symbol bleed between stocks
    ns = {}
    exec(job["formula_code"], ns)
    signal, new_state = ns["evaluate"](df, prev_state)

    _state_store[ticker] = {"_version": formula_meta["version"], "state": new_state}
    return {"signal": int(signal), "new_state": new_state, "error": None}
Imports inside formulas An AI-generated formula can import additional libraries inside evaluate(). This works because Python's import system caches modules — second import of scipy is effectively free. But undeclared imports of unavailable libraries will raise ImportError, caught by the outer try/except and returned as an error result. Always pre-install all expected libraries in the Dockerfile.

6. Data Layer — SQLite Persistence SQLite

SQLite is the persistence layer only — not the serving layer. It survives container restarts and provides the backfill source on boot. During normal operation it is written to (new hourly candles, signal log) but never read from for evaluation. See Section 4 — RAM Layer for the serving path.

Schema

-- Primary OHLCV cache: 2 years of hourly data per ticker
CREATE TABLE ohlcv (
    ticker    TEXT    NOT NULL,
    timestamp INTEGER NOT NULL,   -- Unix epoch seconds (UTC)
    open      REAL    NOT NULL,
    high      REAL    NOT NULL,
    low       REAL    NOT NULL,
    close     REAL    NOT NULL,
    volume    REAL    NOT NULL,
    PRIMARY KEY (ticker, timestamp)   -- covers upsert + read index
);

-- Last time a signal was sent per ticker (deduplication)
CREATE TABLE last_signal (
    ticker         TEXT    PRIMARY KEY,
    triggered_at   INTEGER NOT NULL,
    formula_id     TEXT    NOT NULL,
    formula_version INTEGER NOT NULL
);

-- Audit log of all signals ever fired (optional, recommended)
CREATE TABLE signal_log (
    id             INTEGER  PRIMARY KEY AUTOINCREMENT,
    ticker         TEXT     NOT NULL,
    triggered_at   INTEGER  NOT NULL,
    formula_id     TEXT     NOT NULL,
    formula_version INTEGER NOT NULL,
    signal_value   INTEGER
);

-- Bootstrap state tracking
CREATE TABLE bootstrap_state (
    ticker       TEXT    PRIMARY KEY,
    backfill_done INTEGER NOT NULL DEFAULT 0,   -- 0 | 1
    oldest_ts    INTEGER
);

The PRIMARY KEY (ticker, timestamp) on ohlcv serves triple duty: uniqueness constraint for upserts, B-tree index for range reads, and conflict target for INSERT OR REPLACE. No separate CREATE INDEX needed.

Single Writer Goroutine

Without WAL mode, SQLite is strictly single-writer. A dedicated writer goroutine is the correct pattern: all other goroutines send WriteRequests over a buffered channel; the writer processes them and sends back acknowledgment via a per-request reply channel.

type WriteRequest struct {
    SQL    string
    Args   []interface{}
    Reply  chan error
}

func runWriter(db *sql.DB, ch <-chan WriteRequest) {
    for req := range ch {
        _, err := db.Exec(req.SQL, req.Args...)
        req.Reply <- err
    }
}

Batch Commit Pattern

Stage 1 issues 50 candle writes per tick. Rather than 50 individual Exec calls, the writer should batch them into one transaction. Stage 1 signals the end of a tick batch, and the writer commits once:

// Writer accumulates until it sees a BatchFlush sentinel
case req := <-ch:
    if req.BatchFlush {
        tx.Commit()    // one commit for all 50 candles
        req.Reply <- nil
    } else {
        tx.Exec(req.SQL, req.Args...)
    }

Bootstrap vs Steady-State

On first run (or when a new ticker is added), the engine needs to backfill 2 years of hourly data before evaluation begins. This is a separate operational mode from the hourly tick loop.

[ Engine Startup ] │ ├─ checkBootstrapState() ── reads bootstrap_state table │ ├── [ Tickers with backfill_done=0 ] │ │ │ ▼ │ [ Backfill Worker ] │ Rate-limited to Alpaca free-tier budget │ Paginated historical fetch (max bars per request) │ Writes via SQLite Writer channel │ Sets backfill_done=1 when complete │ └── [ All tickers backfilled ] │ ▼ [ Start Hourly Tick Loop ]

The backfill worker uses a separate rate-limit budget from the live tick fetcher — they should not compete for the same Alpaca API quota. Two distinct token buckets or a single shared one with priority lanes.

Rolling Window Pruning

The ohlcv table is capped at 2 years per ticker. A low-priority background goroutine runs the prune query weekly:

DELETE FROM ohlcv
WHERE timestamp < unixepoch('now', '-2 years');

Run via the SQLite Writer channel with a PRAGMA optimize; and VACUUM; after to reclaim pages. Schedule during off-hours (e.g. Sunday 02:00 UTC).

7. Process Data Manager (PDM)

The PDM is the resource governor between Stage 2 and the Python worker pool. It exposes a simple Submit(EvalRequest) interface and handles all scheduling and resource tracking internally.

Stage 2 ──chan EvalRequest──► [ PDM ] │ ┌───────────┼───────────┐ │ │ │ [ Worker 0 ] [ Worker 1 ] [ Worker N ] │ │ │ Python Python Python stdin/stdout stdin/stdout stdin/stdout │ chan EvalResult │ ▼ Stage 4

PDM Components

Semaphore

Buffered channel of size N (max concurrent evaluations). A worker acquires before evaluating, releases after. Stage 2 blocks if all workers are busy — natural backpressure.

sem := make(chan struct{}, maxWorkers)

Worker Pool

Fixed slice of PyWorker. Created at startup, never torn down. PDM assigns jobs via a work channel each worker listens on.

workers []*PyWorker  // len = N
workCh  chan EvalRequest

Per-job Timeout

Each evaluation wrapped in context.WithTimeout. Go kills the subprocess stdin write, reads any partial stdout, marks result as error. Worker respawns its subprocess on timeout detection.

ctx, cancel := context.WithTimeout(
    parentCtx, evalTimeout)
defer cancel()

Worker Restart

If a Python subprocess exits unexpectedly (crash, OOM kill), the worker goroutine detects the broken pipe, logs to the error channel, and respawns the subprocess before accepting the next job.

if err := w.proc.Wait(); err != nil {
    errCh <- err
    w.respawn()
}

6. Signal Deduplication

Without deduplication, a stock holding above threshold for 6 consecutive hours fires 6 Discord pings. The cooldown is configurable per ticker (natural fit with the existing override system).

-- Stage 4 checks this before firing Discord
SELECT triggered_at FROM last_signal WHERE ticker = ?;
-- If (now - triggered_at) < cooldown → skip
-- If triggered_at is null or expired → fire and upsert

INSERT INTO last_signal (ticker, triggered_at, formula_id, formula_version)
VALUES (?, unixepoch('now'), ?, ?)
ON CONFLICT(ticker) DO UPDATE SET
    triggered_at = excluded.triggered_at,

7. Error Handling

ErrorDetectionAction
Alpaca API failure Non-2xx response or timeout Emit TickerUpdate with Err set. Stage 2 excludes ticker from this tick. Log to error channel.
Stale candle (timestamp mismatch) Stage 2 staleness check Skip ticker this tick. Log warning to error channel.
SQLite write failure Writer goroutine reply channel Critical — halt the pipeline, log to error channel, attempt graceful shutdown.
Python formula exception error field in JSON response EvalResult marked as error. Stage 4 routes to error channel, not signal channel. Subprocess stays alive.
Python subprocess timeout context.WithTimeout Kill subprocess stdin write. Worker respawns subprocess. EvalResult marked as timeout error.
Python subprocess crash Broken pipe on stdin write Worker detects broken pipe, logs crash, respawns subprocess, retries job once.
Discord POST failure Non-2xx response Exponential backoff, max 3 retries. On final failure, log to error channel with full payload so signal is not lost.
Discord rate limit (429) 429 + Retry-After header Respect Retry-After header. Queue pending signals. Never drop.
What must never be dropped silently A signal that was computed but failed to POST to Discord. Always persist the computed signal to signal_log before attempting the POST — then the audit trail exists regardless of webhook status.

8. Resource Limits

Container Level (Docker)

docker run \
  --memory=2g \
  --memory-swap=2g \
  --cpus="2.0" \
  trading-engine

Hard OOM kill at 2 GB prevents the container from impacting the host. No swap prevents slow memory death — fast kill is better for a trading engine.

Process Level (Go → Python)

// Per-evaluation timeout
ctx, _ := context.WithTimeout(
    parent, 30*time.Second)

// Worker pool size = concurrency cap
// indirectly limits total Python mem
maxWorkers = runtime.NumCPU() / 2

Python subprocess memory is bounded indirectly by capping concurrent workers. Explicit per-process memory limits require setrlimit via a wrapper script if needed.

Config file is mounted into the container at runtime (-v ./config.json:/app/config.json). Changing the ticker list or formula parameters does not require an image rebuild — restart the container.
